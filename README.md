# Веб-приложение для ведения личного бюджета

Проект представляет собой клиент-серверное веб-приложение для персонального учета доходов и расходов.  
Система позволяет регистрировать пользователей, вести финансовые операции, управлять категориями, планировать бюджет по категориям и сравнивать плановые и фактические значения за выбранный месяц.

## Основные возможности

- регистрация и вход в систему;
- хранение учетной записи пользователя;
- указание месяца начала ведения личного бюджета;
- автоматическое создание категорий по умолчанию после регистрации;
- создание, редактирование и удаление пользовательских категорий;
- создание, редактирование и удаление финансовых операций;
- планирование бюджета по категориям;
- сравнение плана и факта;
- просмотр данных по выбранным месяцам;
- изменение параметров учетной записи;
- контейнерный запуск через Docker Compose;
- набор ручных API-тестов для проверки backend.

## Запуск проекта

### 1. Распаковка проекта
Распакуйте архив проекта в удобную папку.

### 2. Переход в корень проекта
Откройте терминал в каталоге, где находится `docker-compose.yml`.

### 3. Первый запуск
Выполните:

```bash
docker compose down -v
docker compose up --build
```

Параметр `-v` рекомендуется использовать при первом запуске и после изменения структуры базы данных, чтобы пересоздать том PostgreSQL.

### 4. Доступные адреса

После успешного запуска будут доступны:

- frontend: `http://localhost:3000`
- backend API: `http://localhost:8000`
- Swagger: `http://localhost:8000/docs`

## Особенности интерфейса

- в верхней части бокового меню отображается имя текущего пользователя;
- на страницах **Главная** и **Планирование бюджета** доступен выбор месяца;
- диапазон выбора месяца ограничен:
  - от месяца начала ведения бюджета;
  - до следующего месяца относительно текущей даты;
- на главной странице в блоке **Последние операции** отображаются:
  - дата,
  - категория,
  - тип,
  - сумма,
  - комментарий.

## Работа с датами

Поле **«Месяц начала ведения личного бюджета»** хранится как дата и выбирается через `input type="month"` на frontend.  
При передаче на сервер значение преобразуется к дате первого числа месяца.

## Авторизация

Для защищенных API-маршрутов используется JWT-токен.

### Как это работает
1. Пользователь проходит регистрацию.
2. Пользователь выполняет вход.
3. Backend возвращает `access_token`.
4. Frontend сохраняет токен в `localStorage`.
5. При последующих запросах frontend добавляет заголовок:

```text
Authorization: Bearer <access_token>
```

Если токен отсутствует или невалиден, backend возвращает `401 Unauthorized`.

## Обработка ошибок

### На frontend
Пользователю отображаются сообщения об ошибках:
- при неверном вводе данных;
- при несовпадении паролей;
- при превышении допустимой длины пароля;
- при ошибках API.

### На backend
Сервер обрабатывает типовые ситуации:
- `400 Bad Request` — некорректные данные или конфликт;
- `401 Unauthorized` — нет токена или введены неверные учетные данные;
- `404 Not Found` — объект не найден;
- `500 Internal Server Error` — внутренняя ошибка приложения.

### Ограничение bcrypt
Для хэширования пароля используется bcrypt, который имеет ограничение длины входных данных.  
В проекте добавлена проверка длины пароля до 72 байт как на клиентской, так и на серверной стороне.

## Типовые проблемы и способы решения

### 1. Белый экран на frontend
Если в браузере появляется пустая страница, проверьте консоль разработчика (`F12` → Console).  
В текущей версии адрес backend зафиксирован в `frontend/src/api/client.js` и не использует `process.env`, чтобы избежать ошибки `process is not defined`.

### 2. `401 Unauthorized`
Причина:
- пользователь не вошел в систему;
- токен отсутствует или устарел.

Решение:
- перейти на `/login`;
- выполнить вход заново;
- при необходимости очистить `localStorage`.

### 3. `ERR_EMPTY_RESPONSE`
Обычно означает, что backend упал во время обработки запроса.  
Проверьте логи:

```bash
docker compose logs -f backend
```

### 4. Ошибка PostgreSQL при старте
Если менялась схема БД или база была создана с ошибкой, выполните:

```bash
docker compose down -v
docker compose up --build
```

### 5. Порт уже занят
Если не запускаются `3000`, `8000` или `5432`, проверьте, не используется ли порт другим приложением.

## Ручные автоматизированные API-тесты

В проекте есть папка `tests_manual`, содержащая отдельные Python-скрипты для тестирования backend API.

Важно:
- тесты не запускаются автоматически;
- каждый скрипт запускается отдельно вручную;
- скрипты создают тестовые данные в базе.

### Установка зависимостей для тестов

```bash
pip install -r tests_manual/requirements.txt
```

### Доступные тестовые скрипты

```bash
python tests_manual/test_auth.py
python tests_manual/test_users.py
python tests_manual/test_categories.py
python tests_manual/test_operations.py
python tests_manual/test_budget.py
python tests_manual/test_dashboard.py
```

### Что проверяют тесты

- `test_auth.py`
  - регистрация;
  - повторная регистрация;
  - вход с корректными и некорректными данными;
  - доступ без токена.

- `test_users.py`
  - получение профиля;
  - обновление данных пользователя.

- `test_categories.py`
  - наличие категорий по умолчанию;
  - создание, изменение и удаление категории.

- `test_operations.py`
  - создание, получение, изменение и удаление операции.

- `test_budget.py`
  - создание лимита;
  - получение бюджета;
  - получение сравнения плана и факта.

- `test_dashboard.py`
  - получение данных за выбранный месяц;
  - проверка доступности месяца начала бюджета.

### Изменение адреса backend для тестов

При необходимости можно указать другой URL:

```bash
BASE_URL=http://localhost:8000 python tests_manual/test_auth.py
```
